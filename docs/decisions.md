# Архитектурные решения (ADR)

Фиксирует *почему* приняты ключевые решения — чтобы не пересматривать их заново и не предлагать уже отклонённые альтернативы.

---

## ADR-001: Feature-Sliced Design (FSD)

**Решение:** FSD вместо «по типу» (components/, hooks/, utils/) или доменных модулей.

**Почему:**
- Масштабируется без превращения в «всё в одном месте»: каждый слой отвечает за одно
- Строгие правила импорта (`app → ... → shared`) предотвращают спагетти-зависимости
- AI-ассистент понимает структуру без объяснений — FSD широко задокументирован

**Отклонено:**
- «По типу» (components/, stores/, composables/) — при росте превращается в свалку
- Доменные монолиты без слоёв — циклические импорты неизбежны

## ADR-002: Zod + VeeValidate для форм

**Решение:** Zod (схема + типы) + VeeValidate (интеграция с Vue reactivity) вместо FormKit, Vuelidate, или кастомной валидации.

**Почему:**
- Zod: runtime-валидация + вывод TypeScript-типов из одной схемы, без дублирования
- VeeValidate: нативная интеграция с `<Field>`, поддержка `toTypedSchema(zodSchema)`
- Связка задокументирована, широко используется, активно поддерживается
- Примитивы в `shared/lib/validation` позволяют переиспользовать правила без копипасты

**Ограничения:**
- `zod.default()` конфликтует с `toTypedSchema` — `initialValues` хранятся отдельно
- FormKit мощнее, но тяжелее и сильнее «захватывает» UI-слой

## ADR-003: Ant Design Vue как базовая UI-библиотека

**Решение:** Ant Design Vue вместо Naive UI, Element Plus, PrimeVue, или headless (Radix Vue).

**Почему:**
- Богатый набор компонентов: таблицы, даты, дропдауны, уведомления — всё из коробки
- Качественная документация и активная поддержка Vue 3
- The* компоненты — тонкие обёртки: можно заменить библиотеку без переписывания фич

**Отклонено:**
- Naive UI — хорошая альтернатива, но меньше экосистема
- Headless (Radix Vue) — гибко, но требует реализации всех стилей с нуля

## ADR-004: Axios вместо fetch

**Решение:** Axios для HTTP вместо нативного `fetch`.

**Почему:**
- Интерцепторы: единое место для токена, 401-retry, подавления уведомлений об ошибках
- `withCredentials: true` и `baseURL` работают проще, чем в fetch
- Отмена запросов через `AbortController` работает с обоими, Axios не хуже

**Отклонено:**
- `fetch` — потребовал бы обёртки того же объёма, что и Axios, но без экосистемы интерцепторов

## ADR-005: Pinia вместо Vuex

**Решение:** Pinia (composition API стиль) вместо Vuex 4.

**Почему:**
- TypeScript из коробки — нет mutations, нет verbose boilerplate
- Composition API стиль — тот же паттерн, что во всём остальном проекте
- DevTools поддерживаются так же хорошо

**Конвенция проекта:** actions как `function` (не стрелки) — hoisting + читаемые stack traces.

## ADR-006: openapi-typescript для кодогенерации типов

**Решение:** Генерировать типы из `openapi.yaml` командой `npm run types:generate` → `src/shared/types/api.generated.ts`.

**Почему:**
- Ручная синхронизация типов фронта и контракта API — повторяющийся источник багов
- `openapi-typescript` генерирует точные TS-типы из OpenAPI 3.x без runtime-overhead
- `openapi.yaml` становится единственным источником истины

**Как использовать:**
```ts
// api.types.ts — ручные расширения поверх сгенерированного
import type { components } from './api.generated'
type Budget = components['schemas']['Budget']
```

**Ранее:** типы писались вручную в `api.types.ts`. Подход работал на старте, но SYNC_PLAN.md стал признаком системной проблемы — типы расходились с контрактом.

**Правила:**
- `api.generated.ts` — **не редактировать вручную**, только `npm run types:generate`
- После изменений в `openapi.yaml` на бэкенде — сразу регенерировать и коммитить

## ADR-007: Vitest + Playwright (две системы тестирования)

**Решение:** Vitest для unit/component тестов, Playwright для E2E — вместо одного инструмента.

**Почему:**
- Vitest: нативная интеграция с Vite (тот же конфиг, алиасы, env), быстрый watch mode
- Playwright: реальный браузер, тестирует полный стек включая API мокирование
- Разделение: unit — быстро в CI, E2E — только на критических путях

**E2E покрытие:** только критические пути (auth → транзакция → план). Не каждый компонент.

## ADR-008: Нет SSR

**Решение:** SPA (Vite) без SSR (Nuxt / Vite SSR).

**Почему:**
- Приложение требует аутентификации — публичные страницы для SEO не нужны
- SSR добавляет сложность деплоя (Node-сервер, гидрация, server-only код)
- PWA (vite-plugin-pwa) покрывает offline-потребности

## ADR-009: Токены в памяти (не localStorage)

**Решение:** Access token хранится только в Pinia (memory), refresh token — в `httpOnly` cookie.

**Почему:**
- `localStorage` доступен из JS → уязвимость XSS
- `httpOnly` cookie недоступен из JS → refresh token защищён
- Access token живёт в памяти → не переживает reload, но автоматически восстанавливается через refresh при `restoreSession()`

**Не менять** без обсуждения с бэкендом — стратегия согласована с `SameSite` cookie настройками.
